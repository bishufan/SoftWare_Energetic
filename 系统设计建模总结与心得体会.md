# 系统设计建模总结与心得体会

## 一、项目背景概述

本课题以企业薪资管理系统为对象，完成了一个完整的软件系统设计建模过程。系统采用 Java + Spring Boot 3.2 技术栈开发，涵盖组织架构管理、薪资核算、个税计算、报表统计等核心业务功能。设计建模工作包括系统架构设计、类图建模、顺序图建模、数据库设计、部署建模等多个方面，覆盖了软件工程中系统设计阶段的主要建模活动。

## 二、设计建模工作回顾

### 2.1 系统架构设计

在架构设计阶段，我们采用分层架构加领域模块化的设计思想，将系统划分为三个逻辑层和六个业务模块。基础设施层提供安全、异常处理、跨域等公共能力，传输层统一 API 响应格式，业务模块层按功能领域划分为认证、组织架构、薪资核算、报表和系统管理五个子模块。这种架构设计的优势在于各模块职责清晰、耦合度低，便于团队分工协作和后续扩展维护。薪资核算模块内部又进一步细分为项目定义、核算引擎和工资条三个子模块，体现了高内聚的设计原则。

### 2.2 类图建模

类图建模阶段，我们为系统中的 28 个核心类建立了详细的 UML 类图。建模过程遵循从分析类到设计类的映射方法，将实体类、业务服务类和控制类分别建模。实体类采用 JPA 注解标注持久化映射关系，其中 Department 类的自关联树形结构设计是一个难点，需要在关联关系和序列化之间做好权衡。业务服务类封装了复杂的业务逻辑，如 SalaryCalculationService 中的累计预扣法个税计算算法，涉及多个税率档位的分段计算。控制类作为前端访问的入口，通过 @PreAuthorize 注解实现方法级别的权限控制。

通过类图建模，我们对系统的静态结构有了清晰的认识，每个类的属性、方法和关系都在图中得到了精确表达。类图不仅帮助我们在编码前梳理了设计思路，也成为了后续开发和维护的重要参考文档。

### 2.3 顺序图建模

顺序图建模阶段，我们为核心业务操作绘制了 25 张顺序图，涵盖了部门管理、员工管理、工资核算、报表生成等主要业务流程。每张顺序图详细描述了对象之间的消息传递顺序和协作关系，从用户发起请求到 Controller 接收、Service 处理、Repository 持久化、再到响应返回的完整链路。

顺序图建模的价值在于帮助我们发现了对象交互中的潜在问题。例如在工资核算流程中，SalaryCalculationService 需要依次调用 TaxService、SocialSecurityService 和 SalaryItemService，如果调用顺序不当会导致计算结果错误。通过顺序图的可视化表达，这些依赖关系变得清晰可见。同时在绘制顺序图的过程中，我们也发现了一些接口设计上的不足，比如某些操作缺少必要的参数传递，在建模阶段就及时进行了调整。

### 2.4 数据库设计

数据库设计阶段，我们完成了 8 张关系表的详细设计，共计 84 个字段。设计过程中遵循了数据库规范化原则，每张表都有明确的主键和时间戳审计字段。Department 表通过 parent_id 自关联实现树形结构，这种设计在灵活性上优于固定层级方案，但查询时需要使用递归或应用层处理。SalaryRecord 表作为核心业务表，承载了工资核算的所有关键数据，包括应发工资、个税、社保、公积金、实发工资以及累计预扣法所需的累计收入、累计扣除等字段。在字段命名上，我们避免了使用 SQL 保留关键字，例如将 year 改为 year_val，这是一个在实践中获得的教训。

### 2.5 部署建模

部署建模阶段，我们使用部署图描述了系统的物理部署架构，将系统划分为客户端层、应用服务器层和数据存储层三个层次。部署图明确了各层的技术选型和通信协议，为后续的运维部署提供了指导。

## 三、技术难点与解决方案

### 3.1 Hibernate 懒加载与序列化冲突

在开发过程中，我们遇到了一个典型的技术问题：当 spring.jpa.open-in-view 设置为 false 时，Hibernate 会话在 Service 方法返回后即关闭，而 Jackson 序列化发生在 Controller 返回响应时，此时访问延迟加载的关联对象会抛出 LazyInitializationException。这个问题在部门管理和员工管理功能中表现得尤为突出。

解决方案有两个方向：一是将 open-in-view 设置为 true，保持 Hibernate 会话直到视图渲染完成；二是在实体类中使用 @JsonIgnoreProperties 注解忽略 Hibernate 的代理类属性。我们最终采用了第一种方案，因为它在不影响代码结构的前提下解决了问题，且是 Spring Boot 的默认行为，经过广泛验证。

### 3.2 前端数据展示与后端数据不一致

另一个值得总结的问题是部门创建成功后前端列表不显示。经过排查发现原因是后端的 getAll 方法使用了 findByParentIsNull 查询条件，只返回根部门，新创建的子部门在列表中不可见。这个问题的教训是：前端展示逻辑和后端数据查询必须对齐，后端的查询条件不能无意识地限制了前端需要展示的数据范围。解决方案是将查询改为 findAll，由前端根据需要进行树形结构的组装展示。

### 3.3 全局异常处理的一致性

系统上线前，不同错误场景下返回的错误提示格式不一致，导致前端无法统一处理。例如数据完整性冲突返回的是 Spring 默认错误页面，而业务校验返回的是自定义 JSON 格式。通过重写 GlobalExceptionHandler，我们为七种不同类型的异常分别注册了专门的处理器，统一了错误响应的格式。同时为 Spring Security 的权限拒绝和认证失败也配置了统一的 JSON 响应处理器，确保前端在任何错误场景下都能收到结构一致的响应。

## 四、建模工具的使用体会

本次设计建模使用了 PlantUML 作为建模工具。PlantUML 的文本化建模方式具有版本控制友好、易于协作、可自动化生成的优点。通过编写简单的文本描述即可生成专业的 UML 图形，大大提高了建模效率。特别是结合 Java 代码生成类图的方式，可以保证代码和模型的一致性。

PlantUML 也存在的不足之处。一是对中文字符的支持需要特殊配置字符集编码；二是某些高级特性（如嵌套节点、特定布局控制）的语法支持有限；三是渲染引擎对语法的兼容性在不同版本之间存在差异。这些限制在处理复杂的部署图时表现得比较明显，需要在语法表达上进行一定的折中。

## 五、心得体会

### 5.1 设计建模的价值

通过完成本次系统设计建模，我深刻体会到设计建模在软件开发中的不可替代的价值。模型是思维的载体，将抽象的设计思路转化为可视化的图形，不仅有助于设计者自身理清思路，也便于团队成员之间的沟通和协作。在编码之前进行充分的建模，可以在早期发现设计缺陷、降低后期返工成本。

### 5.2 分层架构的实践感悟

分层架构在实践中面临的一个核心挑战是层与层之间的边界划分。划分过细会导致接口膨胀、性能下降，划分过粗又会导致耦合度增加、复用性降低。在本次项目中，我们采用了领域驱动的模块划分方式，每个模块内部自包含完整的 Controller、Service、Repository 三层，模块之间通过接口进行松耦合的协作。这种划分方式在实践中表现良好，各模块可以独立开发和测试。

### 5.3 理论与实践的结合

在建模过程中，课堂上学到的软件工程理论得到了实际应用。类图中的继承、关联、依赖关系，顺序图中的消息传递、生命线、激活条，数据库设计中的范式理论、索引策略，这些理论知识在建模过程中都找到了具体的落脚点。理论指导实践，实践反过来也加深了对理论的理解。

### 5.4 持续改进的认识

系统设计是一个迭代的过程，不可能一步到位。在本次项目中，部门管理和员工管理功能先后经历了多次修改，每次修改都是基于实际运行反馈的持续改进。这让我认识到，软件设计应该追求在当前的约束条件下做出合理的设计决策，同时保持设计的可扩展性，为后续的迭代优化留出空间。完美的设计不是一次完成的，而是在持续交付和反馈中逐步逼近的。
